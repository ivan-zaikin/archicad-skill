# Типовые документы российской практики (СПДС)

Формы из ГОСТ 21.501-2018 (раздел АР) и СП 54.13330 (жилые здания) и как
собрать данные для них из Archicad 25. Имена свойств проверены на живом API.
Готовый документ оформляй в xlsx/docx (есть отдельные скиллы) — здесь только
выгрузка и расчёт данных.

Все площади — м², объёмы — м³, размеры — метры. В выгрузках встречаются
служебные зоны/элементы (болванки) — показывай разбивку и согласуй
исключения с пользователем.

## Экспликация помещений (ГОСТ 21.501-2018, форма 2)

Графы: номер помещения, наименование, площадь, категория (для производственных).

```bash
python scripts/ac.py values Zone Zone_ZoneNumber,Zone_ZoneName,Zone_NetArea,Zone_ZoneCategoryCode,General_BottomElevationToProjectZero,General_BottomElevationToHomeStory > zones.json
```

Этаж = разность двух последних отметок (см. recipes.md «Этажи»). Группируй по
этажу, сортируй по номеру помещения. Площадь округляй до 0,01 м² (или 0,1 —
спроси пользователя). Итог по этажу и общий итог обязательны.

## Ведомость отделки помещений (ГОСТ 21.501-2018, форма 1)

Графы: наименование помещения, вид отделки потолка/стен/низа стен, площадь.
Из модели берутся площади, виды отделки — от пользователя или из
пользовательских свойств зон.

```bash
python scripts/ac.py values Zone Zone_ZoneName,Zone_NetArea,Zone_WallsSurfaceArea,Zone_WindowsSurfaceArea,Zone_DoorsSurfaceArea,Zone_NetPerimeter > finish.json
```

- Потолок ≈ `Zone_NetArea` (для горизонтальных потолков).
- Стены под отделку = `Zone_WallsSurfaceArea` − `Zone_WindowsSurfaceArea` −
  `Zone_DoorsSurfaceArea` (площадь стен зоны уже без проёмов в части
  поверхностей — сверь на одном помещении вручную, поведение зависит от
  настройки зон в проекте).
- Группируй одинаковые помещения (одно имя + одинаковая отделка) в одну строку
  с суммой площадей — так требует форма.

## Спецификация заполнения проёмов (окна, двери)

Графы: позиция (марка), обозначение (ГОСТ/серия), наименование/размер,
количество по этажам/маркам стен, масса, примечание.

```bash
python scripts/ac.py values Window WindowDoor_MarkerText,General_Width,General_Height,WindowDoor_WHSize,General_BottomElevationToProjectZero,General_BottomElevationToHomeStory > windows.json
python scripts/ac.py values Door WindowDoor_MarkerText,General_Width,General_Height,WindowDoor_WHSize,General_BottomElevationToProjectZero,General_BottomElevationToHomeStory > doors.json
```

Габариты проёма — `General_Width`/`General_Height` (свойств
`WindowDoor_Width/Height` в AC25 нет); `WindowDoor_WHSize` — готовая строка
«ш×в». Марка — `WindowDoor_MarkerText` (как в маркере на плане; например ОК-1,
Д-1). На практике марки часто не проставлены или равны «<Специальный Текст>» —
тогда группируй по (ширина × высота) и предложи пользователю условные позиции.
Размер в форме — миллиметры (умножай на 1000). Считай количество по маркам
и по этажам (этаж — через разность отметок).

## Квартирография / ведомость площадей квартир (СП 54.13330)

Для жилых домов: по каждой квартире — жилая площадь, площадь квартиры,
общая площадь с понижающими коэффициентами летних помещений:
лоджия × 0,5; балкон и терраса × 0,3; веранда и холодная кладовая × 1,0.

Данных «какая зона в какой квартире» в API напрямую нет. Рабочие варианты
привязки (выясни у пользователя, как устроен проект):
1. Номер квартиры в `Zone_ZoneNumber` или в составе номера (напр. «12.3» —
   квартира 12, помещение 3).
2. Пользовательское свойство зоны («Номер квартиры») — читай через
   `GetPropertyIds` с `{"type": "UserDefined", ...}`.
3. Категория зоны (`Zone_ZoneCategoryCode`) различает жилые/летние помещения —
   в виденных проектах: жилые ~1001, СУ ~1002, лоджии ~1004 (уточни по проекту:
   выгрузи пары категория+имя и покажи пользователю).

Жилая площадь = сумма жилых комнат (спальни, гостиные — по именам/категориям).
Площадь квартиры = все отапливаемые помещения. Общая = площадь квартиры +
летние с коэффициентами. Всегда показывай, какие зоны куда отнесены.

## Технико-экономические показатели (ТЭП)

Типовой состав: площадь застройки, общая площадь здания, строительный объём
(в т.ч. ниже отм. 0,000), этажность, количество квартир.

Из модели достоверно считаются:
- **Общая площадь здания** — сумма `Zone_NetArea` по этажам (или площади
  перекрытий по этажам — согласуй методику с пользователем, СП 118 считает
  по внутренним поверхностям наружных стен).
- **Количество этажей** — уникальные отметки этажей (recipes.md «Этажи»).
- **Количество квартир/помещений** — из квартирографии.

Приближённо (предупреждай пользователя об этом явно):
- **Площадь застройки** — площадь перекрытий/полов на нижней надземной
  отметке; точное значение по СП — по внешнему обводу цоколя, из модели
  напрямую не выводится.
- **Строительный объём** — по СП 118 считается по наружным обмерам; сумма
  `General_NetVolume` элементов его НЕ даёт. Оценка: сумма (площадь этажа ×
  высота этажа) по наружным габаритам. Если в модели есть морф/штриховка
  контура здания — используй её.

## Экспликация полов (ГОСТ 21.501-2018, форма 4)

Графы: номер помещения, тип пола, схема/состав, площадь. Площадь —
`Zone_NetArea`; тип и состав пола — из composite перекрытия под зоной
(`GetElementsRelatedToZones` с `elementTypes: ["Slab"]`, затем
`Construction_CompositeName` и слои через `GetComponentsOfElements`)
или от пользователя.

## Ведомость объёмов работ (бетон, кладка)

См. recipes.md «Ведомость объёмов»: `General_NetVolume` по типам элементов
с группировкой по структуре и материалу; чистый объём отдельного слоя
(несущий бетон в composite) — через компоненты. Перед выдачей итогов покажи
разбивку и исключи служебные элементы.
